- 紀律:該走 PENDING_BUSINESS_LOGIC 流程、由 PM 跟客戶確認、Engineer 不擅自加 filter

### L33:User 裁決「先對齊」需精確解讀
- 場景:user 說「先照著 Search_1 查 In Out 對齊這樣」

- 正確解讀:對齊 Suite In/Out 結構 + 明示業務不一致為 PENDING、不擅自加 filter
- 紀律:user「先 X」型裁決默認**僅授權結構性修補**、不授權業務語義裁決
- 後續紀律:對「Golden 與客戶原邏輯不一致」場景、Engineer 主動向 user 釐清「是否授權我加 filter 對齊 Golden」、user 明示授權才動

### L34:撈測資明細看特徵 ≠ 業務 spec 證據
- 我撈 6 筆訂單看出「唯一 esmm_status='01' 是 CM..44」
- 這只是**觀察測資特徵**、不是「客戶業務語義 = esmm_status='01'」
- 兩者差別:測資特徵是 sample observation、業務 spec 是 domain rule、後者需客戶權威
- 紀律:測資觀察可用於**輔助懷疑**(列入 PENDING)、不可用於**直接修法**
