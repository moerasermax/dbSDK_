## 本 sprint 關鍵學習點

### L28：DDD/Clean Architecture 在「null 參數預設行為」議題的層次歸屬
- nullable input 參數的「null 時該怎麼解釋」是**業務語義決策**、不是「技術層面的容錯」
- 該決策屬 Application/BLL layer 邊界、不該:
  - 推給 Domain (Domain 是 data shape、不負責解釋)
  - 推給 DAL (DAL 應 dumb、保持「傳什麼撈什麼」原則、不該猜 default)
- 本次 PoP 區間 null 處理:`ResolvePoPRange` 放 BLL private helper、業務規則明文「null = main range / 半套 = invalid」

### L29：silent-skip 是危險預設行為
- `OrderInfoQuery` 對 `OrderDateStart/End` null 直接 skip filter — 看似「容錯」、實際是「撈全量」絕對危險
- 客戶可能不知道:若不傳 PoP dates、SDK 預設拿全量結果、與「Period over Period」字面意義完全反向
- 紀律啟示:`if (model.X.HasValue) filter.Add(...)` 這種 silent-skip 模式應該:
  - 要嘛上層 (BLL) 確保不傳 null、由 BLL fallback default
  - 要嘛 DAL 明確 throw 拒絕 null (但會違反「DAL dumb」原則)
- 本次選 BLL fallback (符合 Clean Architecture)、不動 DAL

### L30：跨 session ES stale 污染的隱蔽性
- orders-604 stale 30 筆 (2026-04-23 舊 inttest 殘留) — 從未在當前 session 出現、卻被 wildcard search 撈到
- 與 S33 設計 Seeder 時只清當前 ElasticIndex 同源問題:設計者假設「跨 session 清空」、但 wildcard search 把這假設打破
- 紀律啟示:任何使用 `orders-*` wildcard 路由的 search、Seeder 必須清整個 wildcard 範圍
- ES 8.x `destructive_requires_name=true` 預設禁 wildcard delete — 必須 enumerate by name (`GetAsync("orders-*").Indices.Keys.Select(...) → DeleteAsync(name)`)

### L31：三條偏差相互掩蓋的除錯陷阱 (進階版、續 L27)
- 本次三條:
  - (i) Suite 不傳 PoP → null
  - (ii) BLL/DAL silent-skip → 撈全量
  - (iii) ES stale 殘留 → 全量超過實際資料
- 一條接一條、看似互相補強:Suite 不傳就算了、null 撈全量好像也合理、stale 數據看似只是「多了一些」
- User 親自閱讀 dump 對比 Golden 抓出三條 — Suite assertion 全綠是因為 main 範圍正確、PoP 沒驗
- 紀律啟示:寫 Suite 必逐字比 Golden In + Out 完整對齊 (含 PoP / 任何 optional 參數)、不只比關鍵欄位
